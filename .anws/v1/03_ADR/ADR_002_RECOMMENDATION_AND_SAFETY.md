# ADR-002: 推荐逻辑与内容安全策略

## 状态

Accepted

## 日期

2026-06-26

## 背景

产品面向亲子和学生现场体验。

- 用户不一定知道想玩什么，主流程必须是随机抽卡。
- 活动现场需要低等待、低失败率和低尴尬风险。
- 网站库包含第三方外部网站，本系统无法控制外部网站运行时内容。
- v1 不做实时联网搜索、UGC、评论、登录或后台审核流。

## 决策驱动因素

- 因素 1: 安全过滤必须先于随机推荐。
- 因素 2: 筛选只能提供轻控制，不能变成搜索系统。
- 因素 3: 国内优先模式应降低现场打不开网站的概率。
- 因素 4: 全部模式允许更广泛内容，但必须显式标记“可能需要外网”。
- 因素 5: 首版必须靠人工收录保证内容健康。

## 候选方案

### 方案 A: 人工收录 + 本地 JSON + 安全过滤 + 随机排序

- **描述**: 使用 `sites.json` 保存网站库，先过滤 safeLevel 和 childFriendly，再按筛选排序并随机展示。
- **优点**:
  - 可控、透明、容易测试。
  - 不依赖外部 API。
  - 适合 30-80 条 MVP 内容规模。
  - 容易为每条数据保留安全字段。
- **缺点**:
  - 内容增长需要人工维护。
  - 网站失效需要人工复查。

### 方案 B: 实时联网搜索推荐

- **描述**: 用户抽卡时实时搜索或抓取互联网网站。
- **优点**:
  - 内容覆盖面大。
  - 可发现新网站。
- **缺点**:
  - 儿童安全不可控。
  - 网络失败会破坏现场体验。
  - 结果质量和广告风险难以保证。

### 方案 C: AI Agent 动态生成推荐

- **描述**: 使用 Agent 或 LLM 根据筛选生成推荐结果。
- **优点**:
  - 推荐解释能力强。
  - 后续可做个性化。
- **缺点**:
  - 超出 v1 范围。
  - 需要服务端、密钥管理和内容审核兜底。
  - 现场成本与失败模式增加。

## 决策

采用 **人工收录 + 本地 JSON + 安全过滤 + 筛选排序 + 随机展示**。

推荐顺序规则：

1. 过滤 `safeLevel >= 4`。
2. 过滤 `childFriendly = true`。
3. 网络环境为“国内优先”时，优先展示 `domesticPriority = true` 或 `mayNeedGlobalNetwork = false`。
4. 网络环境为“全部”时，允许展示 `mayNeedGlobalNetwork = true`，卡片显示“可能需要外网”。
5. 内容倾向为“轻松好玩”时，优先 `contentMode = light`。
6. 内容倾向为“有点收获”时，优先 `contentMode = useful`。
7. 在排序分组内随机打乱。

内容安全规则：

- 默认只展示 safeLevel >= 4 且 childFriendly = true 的网站。
- 禁止收录成人、擦边、恐怖惊吓、血腥暴力、赌博抽奖、充值诱导、强社交聊天、陌生人互动、未经审核 UGC 社区、大量广告弹窗、下载 App 才能玩、手机号登录、诱导付费、政治敏感或争议内容。
- `tested = false` 允许进入数据集，但 `/blueprint` 必须安排人工测试任务。
- “用户说想玩什么，AI 智能检索推荐”不进入 v1；后续如要做，应创建新版本并补充服务端、密钥、安全审核和检索质量 ADR。

## 后果

### 正面

- 推荐结果可预测、可审查、可测试。
- 现场网络失败面更小。
- 儿童安全边界明确。
- 筛选保持低复杂度。

### 负面

- 内容丰富度受人工收录速度限制。
- 外部网站内容可能变化，需要周期复查。
- 国内优先不是可访问性保证，只是风险降低策略。
- 后续 AI 智能推荐需要另行设计，不能复用 v1 本地随机推荐假设直接加功能。

### 需要的后续行动

- 在 `/blueprint` 中定义 `sites.json` schema 校验。
- 在 `/blueprint` 中定义首批 30 个网站的数据录入与审核任务。
- 在 `/blueprint` 中定义图片占位和截图策略。
- 在 `/blueprint` 中定义空结果和外网标记测试。
